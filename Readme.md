# Skill 的撰写指南

在 AI Agent 开发中，Skill 是一个被严重低估却又极其重要的概念。它不是一份普通的说明文档，而是一个面向特定任务域的能力封装——通过明确的触发描述、结构化的操作指令和必要的附属资源，让模型在某类任务上表现得更稳定、更专业、更一致。

写好一个 Skill，本质上是在回答三个问题：

1. 这个 Skill 帮助模型更稳定地完成什么任务？
2. 用户在什么表达或语境下，应该触发这个 Skill？
3. 不使用 Skill 时，模型最容易在哪些环节犯错？

如果你正在纠结"如何写出一个结构正确、可触发、可维护的 Skill"，这篇指南就是为你准备的。

## 一、Skill 的本质：三层加载机制

在动手写之前，必须先理解 Skill 的天然分层结构。这决定了"什么内容应该放在哪里"：

### 第一层：Metadata — 始终可见

- **内容**：`name` + `description`
- **作用**：始终参与触发判断，决定 Skill 能不能被正确命中
- **原则**：触发信息必须放在 frontmatter，不能藏在正文里

### 第二层：SKILL.md 主体 — 触发后加载

- **内容**：主流程、关键规则、输出要求、分支策略
- **作用**：Skill 被触发后的核心工作指南
- **原则**：只保留"始终该读"的核心内容

### 第三层：Bundled Resources — 按需读取

- **内容**：`references/`、`scripts/`、`assets/` 等
- **作用**：复杂场景的补充资源
- **原则**：长内容、重复内容、确定性内容放在外部资源

**一句话原则**：触发信息放在 frontmatter，核心工作流放在 SKILL.md，其余内容按需外移。

## 二、从最小到最大：三种模板的选择

Skill 不是越大越好，而是要匹配当前的复杂度阶段。我们提供了三套可直接复用的模板：

### 1. 最小版：先跑起来

**路径**：`templates/minimal-skill-template/`

**适用场景**：

- 先快速验证一个 Skill 是否值得存在
- 任务边界清楚，流程较短
- 暂时不需要额外资源、脚本和模板

**核心特点**：

- 只有一个 `SKILL.md`
- 强调先把 `description` 和主流程写对
- 成本最低，最适合起步

**最小结构**：

```text
skill-name/
└── SKILL.md
```

**最小 frontmatter**：

```yaml
---
name: skill-name
description: 说明这个 Skill 做什么，以及在什么场景下应该优先使用它。
---
```

**最小正文骨架**：

```markdown
# Skill Title

一句话说明目标和适用范围。

## Process
- 第一步做什么
- 第二步做什么
- 遇到例外情况怎么处理

## Rules
- 哪些边界必须注意
- 哪些行为不能做

## Output
- 输出长什么样
- 如果没有固定格式，也要说明输出重点
```

### 2. 标准版：稳定复用

**路径**：`templates/standard-skill-template/`

**适用场景**：

- Skill 已经进入稳定复用阶段
- 有一定背景资料、模板或重复性逻辑
- 需要把正文和外部资源分层

**核心特点**：

- 包含 `references/`、`scripts/`、`assets/`
- 适合大多数业务 Skill
- 在可维护性和复杂度之间比较平衡

**标准结构**：

```text
skill-name/
├── SKILL.md
├── LICENSE.txt
├── references/
│   ├── overview.md
│   ├── variants.md
│   └── examples.md
├── scripts/
│   ├── helper.py
│   └── validate.py
└── assets/
    ├── template.md
    └── sample-output.txt
```

### 3. 最大版：长期演进

**路径**：`templates/maximal-skill-template/`

**适用场景**：

- Skill 已经成为一个持续演进的能力包
- 需要评估、测试、角色拆分或多领域扩展
- 需要面向后续迭代和多人协作

**核心特点**：

- 在标准版基础上增加 `agents/`、`evals/`、`docs/`
- 适合复杂 Skill 的长期维护
- **不建议一开始就默认使用该结构**

**最大结构**：

```text
skill-name/
├── SKILL.md
├── LICENSE.txt
├── references/
│   ├── overview.md
│   ├── domain-a.md
│   ├── domain-b.md
│   ├── rules.md
│   └── examples.md
├── scripts/
│   ├── validate.py
│   ├── transform.py
│   ├── generate_report.py
│   └── utils.py
├── assets/
│   ├── templates/
│   ├── samples/
│   └── static/
├── agents/
│   ├── reviewer.md
│   ├── analyzer.md
│   └── comparator.md
├── evals/
│   └── evals.json
└── docs/
    ├── design-notes.md
    └── changelog.md
```

## 三、核心写作指南

### 3.1 Description 怎么写：触发质量的关键

`description` 是 Skill 能否被正确触发的核心。写差了，Skill 就很难被命中；写得过泛，又容易与其他 Skill 冲突。

**一个合格的 description 至少要包含**：

- 任务是什么
- 什么时候触发
- 用户可能怎么表达
- 哪些模糊语境也应归入本 Skill

**推荐公式**：

```text
用于 [目标任务]。当用户提到 [典型表达 / 典型场景 / 相近需求] 时，应优先使用此 Skill，即使用户没有明确说出 [关键词]。
```

**反例**：

```text
用于处理文档。
```

> 问题：只有功能，没有触发时机；"文档"范围过大，边界不清。

**正例**：

```text
用于撰写和整理结构化文档。当用户提到 PRD、技术方案、设计文档、决策记录、规范说明，或表达出"帮我把想法整理成正式文档"的意图时，应优先使用此 Skill，即使用户没有明确说出"文档模板"或"需求文档"等术语。
```

### 3.2 内容放置规则：别把所有东西都堆在 SKILL.md

写 Skill 时，最常见的问题不是"不会写"，而是"内容放错层"。下面是直接可用的放置规则：

#### 什么必须放在 frontmatter

- Skill 名称
- Skill 解决的任务
- Skill 的触发时机
- 应优先命中的典型语境

#### 什么放在 SKILL.md

- Skill 的目标与范围
- 主流程或分阶段执行步骤
- 关键边界与异常处理规则
- 输出格式要求
- 何时读取哪个外部文件
- 不同输入场景的分支入口

#### 什么放在 references/

- 长篇背景知识
- 多个场景的规则差异
- 多个框架、平台、语言的分支说明
- 示例集合、案例库、补充术语表

**拆分信号**：

- SKILL.md 已经明显变长
- 某一部分内容只在少数场景下需要
- 某段说明已经可以独立成专题文档

#### 什么放在 scripts/

- 文件解析
- 数据清洗
- 结果校验
- 格式转换
- 报告生成
- 批量处理

**判断标准**：

- 这一步是否重复出现
- 这一步是否明确且确定
- 这一步是否用脚本执行比用文字描述更稳定

#### 什么放在 assets/

- 输出模板
- 固定格式样例
- 字体、图标、静态素材
- 协议模板、文档模板、表格模板

**一句话判断**：

- 如果文件本身会被直接拿来生成或辅助生成结果，就优先考虑放进 assets/

#### 什么放在 agents/

只有在复杂 Skill 中才需要 agents/。

- 评审者角色说明
- 分析者角色说明
- 比较者或打分者角色说明
- 多阶段协作中不同子角色的职责定义

如果 Skill 只是单流程任务，不要为了"看起来完整"而增加 agents/。

### 3.3 SKILL.md 推荐骨架

下面给出一份适合作为绝大多数 Skill 起点的正文骨架：

```markdown
---
name: my-skill
description: Explain what this skill does and when it should be used.
---

# My Skill

## Overview
说明这个 Skill 解决什么问题，适用于哪些场景，不适用于哪些场景。

## Process
按阶段说明推荐流程：
1. 先做什么
2. 再做什么
3. 遇到分支时如何判断

## Rules
写关键约束、边界、异常处理和质量标准。

## Output Format
如果输出有稳定结构，在这里给出模板。

## References
说明什么时候需要读取 references/、scripts/、assets/ 中的哪个文件。
```

如果任务很复杂，可以继续扩展：

- `## Variants` — 不同变体场景
- `## Failure Modes` — 常见失败模式
- `## Evaluation` — 评估标准
- `## Tooling` — 工具使用说明
- `## Examples` — 示例

## 四、逐步升级策略：从最小到最大

推荐按以下顺序扩展，而不是一开始建满所有目录：

1. **先写最小可用版本**
   - 只有 SKILL.md
   - 写清 name、description、流程、边界、输出
2. **当正文变长时，再拆 references/**
   - 把长文说明和分领域规则外移
3. **当步骤重复且确定时，再加 scripts/**
   - 把校验、转换、生成类工作沉淀为脚本
4. **当输出依赖模板时，再加 assets/**
   - 不要把整段模板都硬塞在正文里
5. **当需要评估和迭代时，再加 evals/**
   - 用真实提示词检验 Skill 的稳定性
6. **当出现多角色协作时，再加 agents/**
   - 只在复杂场景使用

这符合 YAGNI 原则，也最利于维护。

## 五、验收清单

写完一个 Skill 后，至少检查以下项目：

### 5.1 基础结构

- [ ] 是否存在独立目录
- [ ] 是否存在 SKILL.md
- [ ] frontmatter 是否合法
- [ ] 是否至少包含 name 和 description

### 5.2 触发质量

- [ ] description 是否同时写了"做什么"和"何时用"
- [ ] 是否覆盖典型用户说法
- [ ] 是否避免过短、过泛、不可判定

### 5.3 执行质量

- [ ] 正文是否能指导模型执行，而不是只有背景介绍
- [ ] 是否写清输入、输出、流程和分支
- [ ] 是否写清边界、限制和异常处理

### 5.4 结构质量

- [ ] 长内容是否已拆到 references/
- [ ] 重复步骤是否已沉淀到 scripts/
- [ ] 模板和素材是否已放到 assets/
- [ ] SKILL.md 是否明确说明何时读取外部资源

### 5.5 演进质量

- [ ] 复杂 Skill 是否有 2 到 3 个真实测试提示词
- [ ] 是否能继续扩展而不重写整体结构
- [ ] 是否避免了明显重复和自相矛盾

## 六、常见失败模式

以下问题在撰写 Skill 时最常见：

1. **description 只写功能，不写触发场景** — 导致 Skill 难被命中
2. **description 过于宽泛，与其他 Skill 边界重叠** — 导致调用混乱
3. **SKILL.md 只有概念，没有动作导向流程** — 导致模型不知道怎么执行
4. **所有说明都堆在 SKILL.md，没有资源分层** — 导致上下文过长，核心信息被淹没
5. **把本可脚本化的步骤写成大量重复文字** — 浪费 token 且不稳定
6. **示例太偶然，导致 Skill 过拟合单一案例** — 泛化能力差
7. **规则很多，但没有解释为什么重要** — 模型容易忽略或误解
8. **结构一开始做得过大，后续难以维护** — 维护成本高，实际用不上

---

## 七、最终建议

如果你要新写一个 Skill，可以直接按下面的策略执行：

1. 先写一个最小版本，只保留 SKILL.md
2. 优先把 description 写对，因为它决定能否触发
3. 在正文中只保留"始终该读"的核心流程
4. 只在复杂度真实上升后，再增加 references/、scripts/、assets/
5. 当 Skill 开始进入优化期，再增加 evals/ 和 agents/

**一句话总结**：

- 最小结构解决"能不能用"
- 标准结构解决"稳不稳定"
- 最大结构解决"能不能长期演进"

Skill 的本质不是"写一个 Markdown 文件"，而是"把一类任务的正确触发方式、正确处理流程和必要资源组织成可复用能力单元"。

因此，一个高质量 Skill 必须同时满足三个条件：

- 能被正确触发
- 能指导正确执行
- 能在复杂度上升时继续维护和演进
